# 1 Minute Letter - Strict MVP Scope

**Author:** Binh
**Date:** 2025-12-03
**Version:** 1.0 (MVP)
**Status:** ✅ **IMPLEMENTED** (2025-12-03)
**Source:** Refined from `docs/prd.md`

**Timebox:** 3-5 days (solo developer)
**Objective:** Validate that users feel quiet companionship and emotional autonomy, without emotional dependency, pressure, or habit enforcement.

**Implementation:** Complete - All source code in `web/` directory. See `docs/implementation-plan.md` for details.

---

## MVP Objective

**Core Validation Goal:**
Validate that users experience quiet companionship and emotional autonomy when receiving a letter from their future self, without feeling:
- Emotional dependency
- Performance pressure
- Habit enforcement
- Manipulation

**What We're NOT Validating:**
- Business model viability
- User acquisition
- Retention metrics
- Engagement optimization
- Monetization

**What We ARE Validating:**
- Emotional experience quality
- Autonomy preservation
- Absence of pressure
- Absence of dependency

---

## Must-Have Features (Minimum Viable)

### 1. Letter Generation & Display
**MVP-001:** System generates a short letter from "future self" using AI
- **Why MVP:** Core value proposition - this is the entire product
- **Simplest implementation:**
  - Use OpenAI API or similar with a simple prompt template
  - No personalization needed for MVP (generic "future self" perspective)
  - Generate one letter per session

**MVP-002:** Display letter in simple, readable interface
- **Why MVP:** Users must be able to read the letter
- **Simplest implementation:**
  - Single page web app
  - Minimal typography (system fonts OK)
  - White/light background, dark text
  - No animations, no distractions

### 2. Transparency About AI
**MVP-003:** Clear statement that this is AI, not a human
- **Why MVP:** Ethical requirement - must be transparent
- **Simplest implementation:**
  - Small text at bottom: "This letter is generated by AI, not written by a human"
  - No elaborate explanations needed

**MVP-004:** Brief explanation of how letter is created
- **Why MVP:** Transparency principle
- **Simplest implementation:**
  - Link to simple "About" page or expandable section
  - One paragraph explanation

### 3. Zero-Pressure Experience
**MVP-005:** No notifications, no reminders, no prompts
- **Why MVP:** Core principle - respect user autonomy
- **Simplest implementation:**
  - No push notification code
  - No email system
  - No reminders
  - User must actively visit the app

**MVP-006:** No interaction required after reading
- **Why MVP:** Preserve quiet, passive experience
- **Simplest implementation:**
  - No buttons after letter display
  - No "like", "share", "feedback" buttons
  - Just the letter, that's it

### 4. Simple Access (No Login)
**MVP-007:** Access without login or account creation
- **Why MVP:** Hard constraint + reduces friction
- **Simplest implementation:**
  - Single-page web app
  - No backend authentication
  - No user database
  - Generate letter on each visit (or use localStorage for one letter per day)

## Explicitly Excluded Features (To Protect Emotional Integrity)

### Hard Constraints (Never in MVP)

**❌ NO LOGIN:**
- No user accounts
- No email registration
- No authentication system
- No user database
- **Rationale:** Reduces friction, protects privacy, faster to build

**❌ NO NOTIFICATIONS:**
- No push notifications
- No email notifications
- No SMS notifications
- No browser notifications
- **Rationale:** Prevents pressure, respects autonomy, no habit enforcement

**❌ NO STREAKS:**
- No daily streak counters
- No "days in a row" tracking
- No badges or achievements
- No progress indicators
- **Rationale:** Prevents performance pressure, no gamification

**❌ NO SOCIAL SHARING:**
- No share buttons
- No social media integration
- No community features
- No comments or reactions
- **Rationale:** Preserves private, internal experience

**❌ NO ENGAGEMENT OPTIMIZATION:**
- No analytics tracking (beyond basic page views if needed)
- No A/B testing
- No retention tactics
- No "you've been away" messages
- **Rationale:** Prevents manipulation, respects autonomy

**❌ NO MONETIZATION:**
- No payment systems
- No subscriptions
- No ads
- No premium features
- **Rationale:** MVP is about validation, not revenue

### Additional Exclusions (From PRD)

**❌ NO Letter Storage:**
- No saving letters to database
- No history of past letters
- No search functionality
- **Rationale:** Simplifies MVP, each visit is fresh

**❌ NO Frequency Selection:**
- No user preference for delivery frequency
- Hardcode to "one letter per visit" or "one letter per day via localStorage"
- **Rationale:** Simplifies implementation, still validates core experience

**❌ NO Pause/Stop Controls:**
- No pause button (not needed if no scheduled delivery)
- No stop button (user just stops visiting)
- **Rationale:** Simplifies UI, user controls by not visiting

**❌ NO Data Export:**
- No export functionality
- No data portability features
- **Rationale:** Not needed for MVP validation

**❌ NO Offline Support:**
- Requires internet connection
- No service worker
- No offline caching
- **Rationale:** Simplifies implementation

**❌ NO Accessibility Features (Beyond Basic):**
- No screen reader optimization (beyond basic HTML semantics)
- No high contrast mode
- No keyboard navigation optimization
- **Rationale:** Can add post-MVP, not blocking for validation

## Simplest Possible End-to-End User Flow

### Flow 1: First Visit

1. **User opens web app** (single URL, no landing page needed)
2. **See simple interface:**
   - Title: "1 Minute Letter" (or similar)
   - Brief one-sentence explanation: "A quiet letter from your future self"
   - Button: "Receive a letter" (or auto-generate on load)
3. **System generates letter** (via AI API call)
4. **Letter displays** in readable format
5. **User reads letter**
6. **That's it** - no next steps, no buttons, just the letter
7. **User closes tab/browser** when done

**Total interactions:** 1 (opening the app, or clicking "Receive a letter")

### Flow 2: Return Visit (If Using localStorage)

1. **User opens web app**
2. **System checks:** Has letter been generated today?
   - If yes: Display existing letter
   - If no: Generate new letter
3. **Letter displays**
4. **User reads**
5. **User closes**

**Total interactions:** 1 (opening the app)

### Flow 3: Understanding How It Works

1. **User sees small "About" link** at bottom
2. **User clicks** (optional)
3. **Simple explanation page:**
   - "This is an AI-generated letter from the perspective of your future self"
   - "It's designed to offer quiet companionship, not advice or therapy"
   - "No data is stored. No tracking. Just a moment of reflection."
4. **User closes or goes back**

## Allowed Technical Shortcuts for MVP

### Backend & Infrastructure

**✅ Use Serverless Functions:**
- Vercel Functions, Netlify Functions, or similar
- No need for full backend server
- Handles AI API calls server-side (keep API keys secure)

**✅ Use Static Hosting:**
- Vercel, Netlify, GitHub Pages, or similar
- Free tier is fine
- No database needed

**✅ Use localStorage for Simple State:**
- Store "last letter date" to limit to one letter per day
- Store letter content to avoid re-generation on refresh
- No backend database needed

**✅ Use Third-Party AI API:**
- OpenAI API, Anthropic Claude, or similar
- No need to train custom model
- Use simple prompt template

**✅ No Database:**
- No user data storage
- No letter history
- No analytics database
- Everything ephemeral or in localStorage

### Frontend

**✅ Use Simple Framework or Vanilla:**
- Vanilla JavaScript is fine
- Or minimal framework (Svelte, Preact, or similar)
- No need for React/Vue/Angular complexity

**✅ Use System Fonts:**
- No custom font loading
- System serif font for reading (Georgia, Times, etc.)
- Faster load, simpler

**✅ Minimal Styling:**
- Inline CSS or single CSS file
- No CSS framework needed
- Focus on readability, not design polish

**✅ No Build Process (If Possible):**
- Single HTML file with inline JS/CSS is acceptable
- Or minimal build (no bundling complexity)

### Security & Privacy

**✅ No Authentication:**
- No login system
- No user accounts
- No password management

**✅ Minimal Privacy Policy:**
- Simple one-paragraph statement
- "No data collected, no tracking, no storage"
- Can expand post-MVP

**✅ No GDPR/CCPA Compliance (For MVP):**
- Since no data collection, minimal compliance needed
- Add proper privacy policy post-MVP if needed

### Testing & Validation

**✅ Manual Testing Only:**
- No automated tests needed for MVP
- Test with 5-10 real users manually
- Focus on emotional validation, not technical perfection

**✅ No Error Handling (Beyond Basic):**
- Basic try/catch for API calls
- Simple error message if API fails
- No elaborate error recovery

**✅ No Loading States (Beyond Basic):**
- Simple "Generating letter..." text
- No spinners or animations needed
- Keep it minimal

## MVP Success Signals (Emotional Validation)

### Positive Signals (MVP is Working)

**1. Users Feel Quiet Companionship:**
- ✅ Users describe feeling "understood" or "seen"
- ✅ Users feel a sense of calm or peace after reading
- ✅ Users don't feel pressured to respond or act
- ✅ Users can forget about the app without guilt

**2. Users Feel Emotional Autonomy:**
- ✅ Users feel in control of when they use it
- ✅ Users don't feel "needing" to check it
- ✅ Users can skip days without feeling bad
- ✅ Users describe feeling "free" to use or not use

**3. Users Don't Feel Manipulated:**
- ✅ Users don't feel pressured to return
- ✅ Users don't feel guilty about not using it
- ✅ Users don't feel "addicted" or dependent
- ✅ Users trust the transparency about AI

**4. Core Experience Validated:**
- ✅ Users understand the concept of "future self"
- ✅ Users find value in the quiet, passive experience
- ✅ Users appreciate the lack of interaction requirements
- ✅ Users want to share it with others (organically, not because of prompts)

### Measurement Approach

**Qualitative Validation (Primary):**
- Ask 5-10 users after using it:
  - "How did you feel after reading the letter?"
  - "Did you feel any pressure to use it again?"
  - "Did you feel in control?"
  - "Would you use it again? Why or why not?"

**Simple Quantitative (Secondary):**
- Track (manually or basic analytics):
  - Do users return voluntarily? (Not because of notifications)
  - How many days between visits? (Should be irregular, not daily habit)
  - Do users share it organically? (Without prompts)

**Success Threshold:**
- ≥ 80% of users report feeling "quiet companionship" without pressure
- ≥ 80% of users report feeling "in control"
- ≤ 20% of users report feeling "pressure" or "dependency"
- Users return irregularly (not daily habit pattern)

## MVP Failure Signals (Emotional Harm)

### Red Flags (MVP is Failing)

**1. Users Feel Pressure:**
- ❌ Users feel guilty about not using it daily
- ❌ Users feel "behind" or "missing out"
- ❌ Users describe feeling "obligated" to check it
- ❌ Users feel anxious when they don't use it

**2. Users Feel Dependency:**
- ❌ Users describe "needing" the app
- ❌ Users feel anxious without it
- ❌ Users check it compulsively
- ❌ Users can't imagine not using it

**3. Users Feel Manipulated:**
- ❌ Users don't trust the AI transparency
- ❌ Users feel the app is trying to "hook" them
- ❌ Users notice engagement tactics
- ❌ Users feel the app is "too good" (suspicious)

**4. Core Experience Broken:**
- ❌ Users don't understand the "future self" concept
- ❌ Users find it confusing or unclear
- ❌ Users don't see value in the quiet experience
- ❌ Users want more interaction (missing the point)

### Failure Threshold

**If ANY of these are true, MVP has failed:**
- ≥ 30% of users report feeling pressure
- ≥ 30% of users report feeling dependency
- ≥ 30% of users report feeling manipulated
- Users form daily habit patterns (indicates habit enforcement, not autonomy)

**Action if Failed:**
- Stop and reassess
- Identify what's causing the harm
- Remove or modify features causing issues
- Re-test with smaller group
